Git Revision Notes

Version Control Systems (VCS)

A VCS allows a developer to manage the changes made to different files and keep track of who and when any changes were made.

They also allow previous versions of files to be restored if needed.

This makes them especially good at helping groups of people to work together on the same project.

Git is an example of a VCS that is used by many developers.

Using Git in the terminal

The terminal is an interface between the user and their computer's operating system.

It is operated by entering structured commands and viewing the response that is then printed.

The terminal is opened in a specific place in the operating system, such as a folder - this is known as the current working directory.

What is Git used for?

Git is used to create a repository in your computer to keep track of files and changes that are made to them.

It also allows files to be "branched", so that they can be worked on without affecting the main copy.

The branch can then be merged back into the main branch when the updates have been made and tested.

How to use Git

  1. Initialize Git on a folder in your computer:
    This is the folder with the files you want to track in and will make it a local repository.
    Git creates a hidden folder within this folder.
  2. Modify a file within the repository:
    This can be adding a folder, deleting one or changing the content within it. This is done in the working directory.
  3. Select the modified files that you want to stage:
    This means that you want to save the state they are currently in, so you can review the changes or to restore later.
    The files are moved from the working area to the staging area.
  4. Commit the staged files to the repository:
    A permanent record of the files and their contents are saved to the local repository.
    A history of every commit can be seen with Git.
  5. If needed select a previous commit to restore a file:
    Note: Git does not store a copy of the file, but a list of the changes

A list of the main Git commands:

git init

Initialises a new repository

Sets up the intrastructure in the folder to become a repository, including a .git folder.

It also tells Git that it needs to keep track of any files in the repository.

It only needs to be run once, at the start of each project.


git status

Displays the status of the Git repository.

E.g.the branch you are on, any commits, if there are untracked files etc.


git add filename

git add -- all

Adds changes from the working directory to the staging area.

Tells Git which changes we want to commit.


git diff filename

Shows any differences between the file in the working directory and the staging area.


git commit -m "Message added to commit"

Commits the changes in the staging area to the repository.

Permanently stores a snapshot of the files.

The message helps to remind you and others in your team what has changed in the commit.


git log

git log --oneline

Prints a list of the commits in the repository

It shows when it was commited, who did it and on which branch.

It also has a unique code to identify it.

--oneline, shows a more condensed version of the log.


git command -help

Gives help for different commands


git branch branch name

This will create a new branch.

It should have a name that reflects the changes that will be made in it.


git checkout -b branch name

This will create a new branch and switch the terminal to it.


git switch branch name

Switches the repository to an existing branch.


git branch -d branch name

Deletes a branch after the files have been merged


git branch

Lists all the branches in a repository.

There will be a * before the name of the current branch.


git merge branch to merge in

To merge one branch to another one.

Checkout to the branch you want to merge into first, then merge in the other branch.

Git will add changes from branch to merge in into the current branch.

If the branch is not going to be used again, it can be deleted to keep the repository tidy.

If Git can't merge the branches because there is a conflict it will give an error message, explaining where the conflicts are.

They will need to be resolved before merging again.


git revert refnum

Revert back to a previous commit.

Will keep all previous commits etc. intact.


git reset commit 6digituniqueref

Move the whole repository back to a previous commit

Will delete all other commits etc.


git checkout HEAD filename

Rolls back all changes that have benn made to the filename since the last commit.


git --amend -m "Comment"

Modifies the most recent commit.


git remote -v

Returns the name of the remote repository the current local repository is tied to.


git push origin branchname

Pushes the commited changes in branchname to the remote respository.

It will then be visible in GitHub.


git clone URL foldername

Creates a local copy of a remote repository.

The foldername is the folder where the files will be copied to.


git fetch

Fetch downloads items from the remote repository, but they are NOT merged.

The changes will be stored in a new branch in the local repository.

They can be merged with git merge branchname.


Branching

Branching the repsoitory means creating a separate line of development, so you can work on different versions of the same file at the same time.

There can be many branches in the same repository.

Changes can also be made without affecting the main branch of the project and then merged back in.


The default branch is called master and it is created automatically, when you make the first commit.

When there are multiple branches there is a pointer called HEAD, which is the commit you are currently on.

If this is NOT the main branch, then changes are not in sync with the main branch.

To reflect these changes in the main branch, they need to be merged back in.


To push a branch to GitHub, make sure all changes are commited in the local repository, use the command:

git push origin branchname

The new branch should then be visible in GitHub.

To merge the branch into the main branch, create a pull request to merge them.


REMEMBER: The main branch should always be deployable.



Git and GitHub

GitHub is different to Git, but it does use the Git VCS.

It is a remote repository on a network and used as a backup or when working with a team.


Using GitHub

Pushing

Once you have set up a repository in GitHub for your project, copy the URL on the main page.

In the terminal on your computer, when you are in your local repository:

git remote add origin URL

This tells your computer that you are adding a remote repository as an origin to your local Git repository.


To push your main branch from the local repository to GitHub:

git push --set - upstream origin master

Where "origin" is the shorthand name for the remote repository.

Once the code has been pushed to GitHub it can be edited using the code editor in GitHub.

Pulling

This is a combination of two commands: fetch and merge.

Fetch: Gets the change history of a repository or branch. Git log will show the changes.

Merge: Combines the current branch with the branch pulled in from GitHub.

To do them at the same time:

git pull origin

Creating a branch in GitHub

Click on the "master" branch button.

You can then type in the name of the new branch to create it on GitHub.

You can now make changes to files in this branch in GitHub.

To pull this into your local repository:

git pull origin

Then use:

git checkout branch

This will switch to the new branch. We will see any changes made in GitHub in our local files.

Pull Requests

Pull requests notify people of commited changes that are ready to review.

It creates a discussion area so other team members can add comments and discuss the updates before it is merged to the main branch.

Fork

You can't add code to someone else's repository, but you can fork their repository and made a copy of it.

You can also make a copy of it on your local Git, by pasting the original repositories URL:

git clone URL
